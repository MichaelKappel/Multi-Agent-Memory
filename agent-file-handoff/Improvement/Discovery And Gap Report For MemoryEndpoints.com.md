## Risks and blockers

The largest practical blocker is **hosting capability uncertainty**. cPanel supports Python web apps through Passenger/Application Manager or Python WSGI workflows, but actual behavior depends on what the host has installed and enabled, and Passenger uses the newest installed runtime unless manually configured otherwise. If the hosting account lacks Setup Python App / Passenger access, the deployment shape changes immediately. That constraint must be verified before choosing the Python entrypoint and process model. ŅłĆciteŅłéturn16search0Ņłéturn16search3Ņłéturn16search6Ņłéturn16search7Ņłü

A second blocker is the **no third-party runtime dependency** rule. That project constraint implies a standard-library-first Python server approach and browser-native TypeScript/HTML5 in the first pass. It narrows the feasible architecture substantially: no framework assumption, no ORM assumption, no package-managed front-end runtime assumption, and no validator that depends on non-approved packages. The first implementation must therefore stay simple and explicit.

A third blocker is **database initialization and naming discipline**. cPanelŌĆÖs database tooling can prepend the account username as a prefix when database prefixing is enabled. Even though the provided target database name and user already look prefix-qualified, the local hostŌĆÖs actual cPanel behavior still needs to be checked before wiring configuration or writing bootstrap SQL. The first pass should assume a manual database/user/privilege verification step and store only redacted connection metadata in project memory. ŅłĆciteŅłéturn16search8Ņłéturn16search11Ņłü

A fourth blocker is **FTP deployment fragility**. cPanel documents passive-mode behavior and common passive FTP failures, especially around directory listings, timeouts, and firewall/passive-port rules.
